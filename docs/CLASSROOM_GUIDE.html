Autonomous Drone Mission Simulator

Classroom Guide

The classroom build lets one instructor watch a room full of students fly the same mission at the same time, live, on a local network. This guide covers two things: how instructor access is actually controlled, and how to run a class from a cold laptop. Everything in it was checked against the code in this repository.

Section 1How instructor access is controlled

Two things have to be true before anyone reaches the instructor console. Both are checked in src/main.tsx, in resolveRoot().

GateWhat it isWhere
Build flag The bundle must be compiled with VITE_CLASSROOM_ENABLED=true. Without it the whole classroom module is tree-shaken out of the build. src/main.tsx
vite.config.ts
URL parameter The address must carry ?coordinator=1 (instructor) or ?join=<CODE> (student). Neither is present on a normal load. src/main.tsx

The build flag is not a runtime toggle. It decides what code exists in the shipped files. Verified by building both ways:

BuildClassroom chunkAny WebSocket code
vite build none emitted none anywhere in the output
vite build --mode classroom ClassroomEntry-*.js, 50.63 kB (18.87 kB gzipped) only inside that one chunk

So the mobile and Windows builds cannot be talked into classroom mode. There is no URL, no flag, and no console command that turns it on, because the code is not in the file. A unit test keeps it that way (src/tests/classroomBundleGuard.spec.ts).

Say this accurately

There is no account, login, licence key, or credential required to open the instructor console. Anyone holding a classroom-enabled build who types ?coordinator=1 is an instructor. ClassSetup.tsx and classroomClient.ts contain no auth check of any kind.

The commercial gate is therefore who you hand a classroom-enabled build to, not accounts you provision. That is a real gate — it is enforced at build time and cannot be bypassed at runtime — but it is distribution control, not account control. Do not describe it as per-instructor licensing.

Students need no credential either, and that is deliberate. The 6-character class code is a room name, not a secret. Anyone on the same network who knows the code can join as a student. Nobody, including the relay server, can read what a student does without the instructor's key.

Section 2What the encryption guarantees

This part is strong and can be stated confidently.

  • End-to-end encryption. x25519 key agreement, HKDF-SHA-256 to derive the session key, AES-256-GCM per message with a fresh IV (src/classroom/sessionCrypto.ts).
  • The relay only sees ciphertext. It routes on the plaintext envelope — message type, class code, student id — and forwards the sealed payload untouched. It never parses or opens it. It cannot read a single student metric (server/classroom.mjs).
  • The instructor's private key never leaves their browser. It is generated in the tab, held in memory, and never sent anywhere. Nobody can recover it for you, and there is no vendor-side copy. If that tab is lost, that class's data is unrecoverable. This is inherent to real end-to-end encryption, not a defect, and the class-creation screen says so on screen.
  • Re-binding a live class costs a token. When a class is created the relay mints a 256-bit instructor token and gives it only to the creating socket. Any later attempt to point that class at a different key must present the token, compared in constant time. Someone on the network who overhears the six characters read aloud cannot seize the room.
  • Live frames are never stored. Only the end-of-run submission is written to disk, and it is written as ciphertext, under classroom-runs/<classId>/ on the instructor's own machine.
  • Replayed frames are rejected. Every sealed message carries a counter inside the ciphertext, under the authentication tag. The instructor keeps a high-water mark per student and drops anything that repeats or rewinds.

The honest summary: the crypto is the real boundary, and it holds. The access gate in front of it is distribution, not identity.

Section 3If you want real per-instructor licensing

This does not exist today. Nothing in the code identifies, counts, or expires an instructor.

The option that fits this product is an offline signed activation key: an Ed25519 signature over the instructor's name and an expiry date, verified in the browser against a public key embedded in the build, using the @noble/curves dependency the project already ships. It needs no server and no network call, so the offline and LAN-only posture survives intact.

Be clear about what it buys. A client-side check is honest friction — it stops casual sharing and makes expiry real — not unbreakable DRM. Anyone willing to patch a JavaScript bundle can remove it.

Section 4One-time setup

On the instructor's laptop only. Students install nothing — they use a browser. You need Node 20 or newer. 

npm install

Two commands run the classroom:

CommandWhat it doesWhen to use it
npm run classroom Builds the classroom bundle, then starts the relay server. First run of the day, and after any code change.
npm run classroom:relay Starts the relay against the bundle that is already built. Restarting the server between classes. Much faster.

The relay serves the built app and relays messages on the same port, so students only ever need one address.

Port

Default is 8080. Override with an environment variable or a first argument; the argument wins. 

node server/classroom.mjs 9000   # port as an argument
PORT=9000 npm run classroom:relay

Section 5Starting a class

  1. Run npm run classroom. On boot the relay prints its address once for localhost and once for every non-internal IPv4 address on the machine:

    Classroom relay on http://localhost:8080
Classroom relay on http://192.168.1.24:8080

    The one that is not localhost is the address students need. If more than one is printed, it is the one on the same network as the students' wifi.

  2. Open the instructor console in a browser on that laptop:

    http://192.168.1.24:8080/?coordinator=1

  3. Pick a scenario from the list. All 21 catalog scenarios are available.

  4. Check the seed. It defaults to the scenario's own seed. Reroll gives a random one. Whatever is showing when you create the class is what every student flies, so all of them get byte-identical conditions — same weather, same comms dropouts, same thermal contacts, same battery pressure. That is what makes comparing their runs fair.

  5. Press Create class. The console appears with a 6-character code in the header, in large type. Read it aloud.

    Codes use digits and consonants only — vowels are removed so a code can never accidentally spell a word.

Do not refresh this tab

The instructor's private key exists only in that tab's memory and is never written to storage. Refreshing or closing it ends the class for everyone and the key is gone. Leave it open for the whole session.

Section 6Students joining

Students must be on the same wifi as the instructor's laptop. They open: 

http://192.168.1.24:8080/?join=<CODE>

The code arrives prefilled. They type their name and press Join class. If they mistyped the code they can correct it on that screen.

As soon as they join, the assigned mission loads and starts. Students do not pick a scenario, run preflight, or plan launch bays — they land straight into the mission the instructor published, already flying, with a small banner at the top of the screen showing the class code. Their tile starts reporting immediately.

Rough edge — you must give out the full address

The console does not display a join URL or a QR code. It shows the code only, so the instructor has to tell students the address as well — write it on the board before class starts.

The plain base address does not work as a fallback. Opening http://192.168.1.24:8080/ with no parameter loads the ordinary simulator, not the join screen. The ?join= part is what reaches the join screen at all.

Section 7Running the class

The console is a wall of tiles, one per student, updating once a second. Each tile is a small live map of that student's mission, not a video — the browser is drawing their drone positions onto a shared picture of the scenario, so twenty tiles cost almost nothing.

What each tile shows

  • The scenario backdrop, drawn once and shared by every tile: geofences in red, search areas in cyan, planned routes in grey, launch and recovery sites as small grey squares.
  • One dot per drone with a heading triangle. Green means flying, amber means returning to base or recharging, red means emergency, stranded, or awaiting recovery, grey means landed, idle, or recovered.
  • Below the map: the student's name, elapsed mission time as T+mm:ss, the number of drones still active, and the lowest battery in their fleet — which turns red under 20%.
  • A coloured border when something needs attention.

There is no status word on the tile. Status is carried by the border colour and the drone colours.

Alerts

Nobody can watch twenty tiles at once, so the wall sorts itself. Any tile with an alert moves to the top-left — critical first, then warning, then everyone else — and the student's name appears as a coloured pill in the strip above the grid. Clicking either the tile or the pill jumps straight to that student.

SeverityBorderRaised by
CRIT red geofence breach · comms lost (signal at or below −90 dBm) · battery critical (under 10%) · emergency · airspace conflict · recovery needed (stranded, recovery requested, or recovery en route)
WARN amber comms degraded (signal at or below −80 dBm) · battery low (under 20%) · returning to base · stalled · idle · new thermal contact

A tile with any critical bit set is red, even if it also has warnings. The three warning bits for stalled, idle, and new thermal contact are defined in the code but are not raised by the current build — see known rough edges.

Above 16 students the wall automatically switches to smaller tiles. There is no layout control to set.

Header

The header carries the class code, the scenario name, the number of students joined, a toggle between the results pane and the focus pane, and End class. An amber N rejected badge appears if any message failed to decrypt or repeated a sequence number; the usual cause is a student who joined a different class code.

Section 8Watching one student

Click a tile. That student's browser starts a second, more detailed stream at roughly three times a second, and the right-hand pane fills with:

  • a larger live map of their mission;
  • mission time, wind in knots, visibility in miles, and their thermal contact count;
  • a per-drone table: drone, mission state, altitude, battery, signal.

One student at a time. Clicking another tile switches the detailed stream to them and stops the previous one. This is a deliberate design limit, not a setting: the extra traffic never scales with class size.

Everyone else keeps streaming their once-a-second tile the whole time, including students whose simulation is paused. A paused student still reports, which is usually exactly when you want to know.

Section 9After the class

When a student's mission ends, their run is encrypted and sent automatically. No action from the student. Submissions appear in the results pane as they arrive.

The comparison table holds one row per student:

ColumnMeaning
Studentthe name they typed at join
Outcomecomplete, ended, or a dash
Dur(s)mission duration in seconds
WPwaypoints reached
Conflictsairspace conflicts detected
Geofencegeofence breaches
RTBreturn-to-base triggers
Thermalthermal contacts
Recoveryrecovery dispatches
MinBatt%lowest battery any of their drones reached
Verifiedwhether that run's evidence hash chain verified

Export CSV downloads the table as class-<CODE>-results.csv. Nothing is scored or graded — these are the numbers the simulator already records.

The relay also writes an encrypted copy of each submission to classroom-runs/<classId>/ on the instructor's machine, as a backup against a crashed browser tab. Those files are ciphertext and the app has no way to import them back today.

If a student flies a second time in the same class, their newer run replaces the older one in the table.

Section 10Limits

These come from src/classroom/limits.json, which the browser and the server both read, so there is one set of numbers rather than two that can drift.

LimitValueWhat happens at the limit
Students per class 40 The next student is refused with "That class is full."
Classes per relay 8 Creating a ninth is refused with "This relay is already hosting its maximum number of classes."
Message size 256 KB Anything larger is dropped by the relay without comment.
Heartbeat timeout 30 s The relay pings every 15 seconds and disconnects anything silent for the full 30.
Class code 6 characters Digits and consonants only, from a fixed 31-character alphabet.

Two more, set on the browser side: tiles update once a second, the focused student about three times a second, and a student whose connection is backed up past 64 KB skips a frame rather than queueing a stale one.

Section 11Troubleshooting

SymptomCause and fix
Students cannot reach the address at all. Almost always the firewall on the instructor's laptop. Windows prompts to allow Node through on private networks the first time — if that prompt was dismissed, allow it in the firewall settings. Then check: everyone on the same wifi, not a guest network; students using the printed LAN address, not localhost; some guest and campus networks isolate clients from each other and will block this no matter what the laptop allows.
"No class with that code is running." Wrong code, or the class was ended. Codes have no vowels — a reported "O" is a zero, and there is no I, A, E, or U. Read the header code again.
"That class is full." 40 students are already in. Start a second class on the same relay.
"This relay is already hosting its maximum number of classes." Eight classes are live on that server. End one, or restart the relay with npm run classroom:relay — that clears all class state, which also ends every class it was hosting.
"That code is already running on this relay." A code collision with a live class. Press Reroll and create again.
The instructor refreshed or closed the console tab. The class is over. The key lived only in that tab and cannot be recovered. Students see "Class ended". Create a new class and read out the new code.
The instructor's wifi dropped for a few seconds. If the tab stayed open, it retries for about 20 seconds and comes back under the same code with the same key. Students will have been told the class closed, so have them reload and join again with that same code.
An amber N rejected badge in the header. Messages arrived that would not decrypt, or repeated a sequence number. Normally a student who joined a different class code. Hover the badge for the breakdown.
?coordinator=1 loads the ordinary simulator instead of the console. The relay is serving a bundle built without the classroom flag — usually because a plain npm run build ran after the classroom build. Run npm run classroom to rebuild and restart.
A student's tile is blank or frozen. Their browser has stopped sending. Have them reload the join URL and rejoin. They keep their name; their earlier run is not lost unless it never finished.

Section 12Deployment

Three builds come off this one codebase, distinguished only by environment variables set per Vercel project:

BuildEnvironment
MobileVITE_APP_TARGET=mobile
WindowsVITE_APP_TARGET=windows
ClassroomVITE_CLASSROOM_ENABLED=true, with VITE_APP_TARGET left unset so phone students get the mobile shell and the instructor gets the desktop console automatically
Vercel cannot host the relay

The classroom client can be hosted on Vercel. The relay cannot. It is a long-lived WebSocket server that holds every class's state in memory for the duration of the class, and Vercel is serverless — there is no process to hold that state.

So the hosted copy exists to show the interface. Real classes run on the local network from npm run classroom. This is settled.

Running on the LAN is also why the classroom works over plain HTTP with no certificates: the encryption is implemented without the browser API that would demand a secure context. Nothing has to leave the building.

Section 13Known rough edges

Things that will be noticed in a first class, stated plainly.

  • No join URL or QR code on screen. The console shows the code only. The instructor has to supply the address separately.
  • The instructor tab cannot be refreshed. The key is not persisted, so a reload ends the class. There is no resume.
  • The "Graded (lock seed)" checkbox does nothing yet. The seed shown at creation is sent to every student either way. The checkbox does not change what is published.
  • Only scenario and seed are selectable. The class always runs at daytime, summer, with the weather, comms, thermal, battery, and terrain dials at zero. The deeper variant settings exist in the simulator but are not exposed on the class setup screen.
  • Three alert types never fire. Stalled, idle, and new thermal contact are defined and unit-tested, but the live publisher does not currently raise them. Everything else in the alert table works.
  • No mission authoring from the console. The class configuration format supports a custom-authored mission, but the setup screen offers the catalog only.
  • Encrypted run backups cannot be re-imported. The relay writes them; nothing reads them back.
  • No instructor notes or rubric scoring. The results table reports simulator metrics only.
