Build isolation — the hard rule this plan obeys

Nothing classroom-related ships in the Mobile or Windows builds. At all. Not the assessment engine, not the rubric, not the scorecard UI, not the command channel, not a type, not a string. Tracks A and D are operator capabilities and land on all three builds; tracks B and C are instructor capabilities and live entirely behind the classroom's one-way door. See §1.3 for the invariant and its enforcement.

The three findings that shape everything

1 · The classroom build is not a third UI. ClassroomEntry.StudentLive renders <App /> verbatim. Put a control in the two shells and the student inherits it — with full authority, since the store defaults operatorRole: 'pic'. Three-build parity collapses to two shells plus one instructor signal.

2 · Roughly 60% of the retask pipeline already exists, scoped to one drone and driven by an if/else ladder. Propose → geofence-audit → diff → accept → hash-chain is built and tested. What is missing is fleet scope, a scoring model, and situational totality.

3 · The classroom crypto layer is already hardened enough to carry commands. Symmetric ECDH session keys, sealed {seq, body} anti-replay, a server-minted instructorToken, and directed instructor→student routing all exist today. The command channel is a mirror of primitives already in the repo — not new cryptography.

1.2 The governing constraints

  • Determinism is the product's central claim. Same seed must produce identical output on Mobile, Windows and Coordinator. Every new module is pure: no Date.now(), no Math.random(), no wall-clock, no I/O.
  • No ML, anywhere. The roadmap bans it explicitly — "breaks determinism, the most defensible claim the project has." Every algorithm here is closed-form scoring plus bounded search.
  • Zero network in src/. Enforced mechanically by an ESLint no-restricted-globals rule over src/sim and src/scenarios, plus the classroom bundle guard.
  • Simulation fidelity is uniform across builds; only rendering differs. Note the scope of this rule carefully — it governs simulation: physics, detection, occlusion, determinism. It does not mandate that a pedagogical scoring layer ship everywhere. See §1.3.

1.3 Build isolation — the one-way door

The classroom is a separate edition, not a feature flag on the other two. The dependency rule is directional and absolute:

  src/classroom/  ──imports──▶  src/sim/ · src/store/ · src/components/     ✅ ALLOWED
  src/classroom/  ──imports──▶  src/App.tsx                                 ✅ ALLOWED

  App / MobileShell  ──imports──▶  src/classroom/                           ❌ FORBIDDEN
  ─────────────────────────────────────────────────────────────────────────────────────
  ONE permitted crossing: main.tsx's flag-gated dynamic import
      VITE_CLASSROOM_ENABLED === 'true'  &&  ?join= | ?coordinator=1
          → import('@/components/classroom/ClassroomEntry')

Verified against the working tree — and already enforced in CI

A sweep of every .ts / .tsx outside src/classroom/ and src/components/classroom/ found exactly one reference to classroom code: the dynamic import in main.tsx. The invariant holds today.

More than that — it is already enforced mechanically, against built artifacts, by scripts/assert-bundle-isolation.mjs, wired into CI as npm run assert:bundles. That script is stronger than a source-level check: it builds both ways and asserts the difference.

default build   → no classroom chunk emitted at all · no networking token anywhere
classroom build → classroom chunk exists · networking confined to it · relay client not dropped

The second half is the part most such guards omit: a check that only inspected the shipping bundle would still pass if the classroom feature silently stopped shipping altogether. This one fails in both directions.

The one hole that guard cannot see — and it is exactly the hole this plan could open

The artifact guard detects leakage by two signals: a chunk filename matching /Classroom/i, and networking tokens (new WebSocket(, WebSocketServer). ClassroomEntry is reached by dynamic import, so it gets its own named chunk and both signals fire.

But a statically imported, non-networking classroom module would trip neither. If App.tsx imported classroom/missionAssessment.ts, Rollup would inline it into index-*.js — no chunk named "Classroom", no WebSocket token — and the guard would report OK while the rubric shipped to every Mobile and Windows user. That is precisely the module this plan adds, so the residual assertion in §G is not redundant polish; it closes the one gap that this specific work opens.

What this costs, stated honestly. The classroom student runs <App />, so they inherit tracks A and D for free — that part is unchanged. But an instructor-only surface such as the scorecard cannot be rendered from inside App, because that would require App to import it. Instead the classroom shell renders it as a sibling overlay around App — precisely the pattern StudentLive already uses for the cls-banner. Same screen, zero coupling.

The objectives / assessment seam

missionObjectives.ts stays in src/sim/ and missionAssessment.ts moves to src/classroom/. The line between them is "what is this mission trying to achieve" versus "how well did you do." Objectives are mission definition — the tactical advisor consumes them directly for its valueGain term, and any solo operator legitimately wants to know how much of the task remains. Assessment is grading: bands, caps, rubric findings, life-safety violations. That is a teaching artifact and it ships only with the teaching edition.

ATactical fleet retask

One press. The advisor reads the mission in totality, scores a bounded candidate set per aircraft, assigns across the fleet, and commits immediately — new coordinates spawn on the map and the aircraft turn on the next tick. An inline undo bar replaces the confirm dialog.

A.1 Execution model — how "autospawn" actually feels

t = 0 ms      plan computed                    synchronous, sub-millisecond
t = 0 ms      N magenta dashed polylines spawn; waypoint pins drop in sequence
t ≈ 250 ms    lines flip to each drone's own colour  = COMMITTED
next tick     aircraft turn onto new headings        ≤ 50 ms

              ⟳ FLEET RETASKED · 6 drones · +18% POD · ↺ UNDO (8s)

No modal, on any shell. The visual is already built — TacticalMap carries a suggested-route source and a magenta dashed line layer today, refreshed on a 100 ms interval. Repurposing it means zero new MapLibre layers and zero new GPU buffers, which keeps the mobile memory rule intact.

Why immediate commit is safe

The sim loop re-reads droneWaypoints from getState() on every physics sub-step, and a React click handler can never interleave with the synchronous sub-step loop. Application always lands cleanly between frames. The advisor reads a snapshot and never ticks physics, so determinism is untouched.

A.2 The algorithm

Step 1 — snapshot the mission in totality. One pure read produces MissionSituation: per-drone position, battery vs reserve, effective drain rate, platform speed/endurance/sensor, signal, mission state, remaining route; environment (wind, speed cap, drain multiplier, sensor confidence, hazards, forced-RTB); objectives (unresolved contacts weighted by confidence × recency, operational features, due dispatch tasks, recharge stations); and swept coverage per sector derived from positionHistory.

Step 2 — bounded candidate generation (≤ 8 per drone) reusing the six existing route patterns, plus hold_station (always present, so the advisor can honestly recommend "keep going"), deep_scan(contact) per top-K contact, route_recharge(station), and rtb_now. Every candidate passes the existing, already-tested geofence audit.

Step 3 — score. Every term is computable from modules that already exist:

TermDerivationRole
valueGainpodFromCoverage + contact confidence × recency + feature prioritywhat the task buys
transitCosthaversineDistanceM ÷ (maxSpeedMs × speedCapMultiplier)time to effect
energyFeasibleroute time × drain × weather multiplier vs batteryPct − reservePcthard gate
riskPenaltygeofence proximity, signal, active hazards, conflict flagsafety
continuityPenaltyfraction of current route abandonedanti-thrash
redundancyPenaltyobjective overlap with another drone's claimmakes it fleet logic

Step 4 — assign. Greedy over pairs sorted by score with a stable (droneId, candidateId) tie-break, then ≤ 2 bounded swap-improvement passes. With D ≤ 8 this is free. The last two scoring terms are what separate a fleet plan from eight independent selfish choices.

The trap that will make it look broken

getNextCommand applies its emergency guards before any commanded route — below-reserve, geofence breach, and weather-forced RTB all coerce the aircraft to return_to_base regardless of what was just commanded. An advisor that ignores those thresholds will emit plans the simulation overrules one tick later: the operator watches a route spawn and then watches the drone turn for home. The advisor must gate on the identical thresholds MissionManager enforces.

B.2 Protocol — three additive message types, no version bump

interface ClassCommandMsg {           // instructor → server → one student (or broadcast)
  v: 1; type: 'class.command'; classId: ClassId;
  studentId: StudentId | null;        // null = whole class
  instructorToken: string;            // server-side authority gate
  sealed: Sealed;                     // SealedPayload<InstructorCommand>
}
interface CommandMsg   { v: 1; type: 'command';     classId: ClassId; sealed: Sealed }
interface StudentAckMsg{ v: 1; type: 'student.ack'; classId: ClassId; from?: StudentId; sealed: Sealed }

PROTOCOL_VERSION stays 1. decodeEnvelope validates version, type and classId only, and rejects unknown types before dispatch — so an older client ignores commands rather than misbehaving. Routing is the identical shape as the existing class.focus → focus.on path.

B.3 Security model — five non-negotiables

  1. Server gates on instructorToken using the same constant-time compare that already guards instructor re-binding. Without it, anyone who overheard a six-character class code could puppet the room.
  2. Outbound sequence mirror. Instructor holds outSeqByStudent; student enforces acceptsSeq through a single decrypt door. Without it, a captured end_mission is replayable forever.
  3. Typed whitelist, validated after decryption. Unknown or malformed commands are rejected and acknowledged as failures.
  4. Rate limit in limits.json, enforced server-side, shared by the TypeScript client and the plain-JS relay.
  5. Integrity counters on the student side too. A student seeing commandRejects > 0 means someone is injecting.

The attribution trap — highest-value finding in this track

MissionEvent carries operatorId and role, and emitEvent fills operatorId from the active operator. If instructor commands flow through the same path unmarked, the assessment engine in Track C will score the student on the instructor's actions. An instructor who RTBs a drone to save it hands the student credit for battery discipline they never exercised.

Fix — with zero classroom leakage into shared types. An earlier draft proposed a EventType: 'instructor_command' member and a MissionEvent.actorKind field. Both live in src/types/index.ts, which every build compiles — so both would put the word "instructor" inside the Mobile and Windows editions. Rejected under §1.3.

Instead: emitEvent already accepts an optional operatorId: string and falls back to the active operator. Add one generic, classroom-agnostic primitive to the store — a commandActorId override (or a withCommandActor(id, fn) scope) that recordOperatorCommand honours. Mobile and Windows never set it, and nothing in the shared code knows what a classroom is. The classroom command executor sets it to a namespaced id, and the classroom-side assessment partitions on that prefix.

Do not add 'instructor' to OperatorRole under any circumstance — that type models a simulated aviation role (PIC / MC / Observer), and conflating a real-world actor into it would corrupt the existing authority gates in addition to breaking isolation. With attribution in place, student events score the student and instructor events are reported as interventions — itself a first-class teaching signal.

B.4 Visible takeover is a design requirement

CLASS 7KX3M2 · streaming · ⬤ INSTRUCTOR CONTROL — "UAV-03 → RTB"          [3s]

The classroom banner already renders "instructor watching." Extend it to flash every executed command. Silent remote control of a person's screen is a trust failure and, in a school deployment, a policy problem. The join gate should state that the instructor can observe and control the session — joining with the code is the consent act. Every command also lands in the student's own event log, so the debrief is honest in both directions.

CAssessment & rubric engine classroom edition only

A pure module that answers three questions continuously, for every student on the wall: how far through the task are they, are they meeting the rubric, and did they put human lives first? It ships with the instructor-led edition and nowhere else — see §1.3. The objectives it measures against are shared sim data; the grading is not.

C.1 Doctrine spine — lexicographic, not weighted

The rubric is built on the ICS / NIMS incident priority hierarchy, the order every US incident commander is trained to work to:

1. Life Safety  →  2. Incident Stabilization  →  3. Property & Environmental Conservation

The load-bearing property is that this order is lexicographic. You do not trade life safety for stabilization efficiency at any exchange rate. Therefore the rubric must be lexicographic too — and this is the entire answer to "ensure the operator puts human lives first."

Why a weighted life-safety term would fail

If life safety is one weighted term in a sum, students optimise the sum and trade lives for coverage points. Rubrics get gamed — that is what rubrics do. So Tier 0 is a gate that caps the achievable total, not a term that contributes to it.

Tier 0  LIFE SAFETY          → PASS/FAIL gate. Any violation CAPS the total.                                     minor → cap 79 · major → cap 59 · critical → cap 39 Tier 1  INCIDENT STABILIZATION → 0–60 weighted, reachable only if Tier 0 passes Tier 2  RESOURCE STEWARDSHIP   → 0–40 weighted

A student who achieves 100% sector POD, flawless battery discipline and a clean recovery — but left a 0.9-confidence thermal contact unactioned for eight minutes — scores 39. That is the lesson, delivered by arithmetic rather than by a comment in a debrief.

Every check is computable today from events, thermalContacts, drones and metrics. No new simulation machinery.

C.3 "% done with task" — the objective model

Placement is load-bearing — and this is a deliberate reversal

An earlier draft argued missionAssessment.ts should live in src/sim/mission/ so all three builds could render a scorecard, citing the "simulation fidelity is uniform across builds" principle. That was an over-application of the rule. That principle governs simulation — physics, detection geometry, occlusion, determinism — so that a seed reproduces identically everywhere. A rubric is not simulation. It is a read-only observer of simulation output that assigns a grade, and it changes no physics whatsoever. Nothing in the fidelity rule requires it to ship to an operator who has no instructor.

Correct placement: src/classroom/missionAssessment.ts, pure and deterministic exactly as before, but inside the classroom's one-way door. It imports downward into src/sim/ for sweepWidth, missionObjectives and the types it scores — which is the allowed direction. The scorecard renders as a sibling overlay beside <App /> in the classroom shell, never from inside App.

Consequence: keep the assessment out of the shared after-action package

buildAfterActionPackage is shared by all three builds. Folding the scorecard into it would drag the rubric straight into the Mobile and Windows bundles through the export path — the exact leak §1.3 forbids, arriving by the back door. The shared AAR export stays exactly as it is. The classroom produces its own debrief artifact from the sealed end-of-run submission, which it already receives and already stores.

Also worth stating plainly: this module is the roadmap's WP-9 (laneScoring.ts, described there as "the one auto-scorable number for the Coordinator dashboard"), generalized from a single NIST lane to all 21 scenarios — and it delivers without waiting on the WP-4 keystone.

DLaunch bay doctrine & mobile stations

Launch bays are currently assigned by array index. They carry no reachability check, no doctrine, no recovery symmetry, and they cannot move — even though three of the nine site kinds in the type system are explicitly mobile platforms.

D.1 What is actually broken today

LB-1Auto-assign is round-robin by array indexblocker

buildAutoAssignments does siteEntries[i % siteEntries.length]. It ignores distance to the drone's first waypoint, site kind, capacity, weather closure, agency, platform endurance — and it ignores droneRouteBriefs[id].launchRationale, which already contains the authored tactical justification for that drone's launch site. The "⚡ Auto-Assign" button can therefore produce a plan that immediately fails computeBlockers for over-capacity or weather closure.

Fill: replace with a scored assignment from launchDoctrine.ts (§D.2) that respects the same blockers it will be validated against.

LB-2Two drones on one bay spawn at identical coordinatesblocker

initFleet resolves explicitBays[id] to the assigned site's position — the same LatLng for both drones. planCoordinatedLaunch then short-circuits: if (explicitBays?.[id]) bay = explicitBays[id], with no intra-site fan. Default capacity is 2 drones per bay, so the standard case stacks two aircraft on one point — defeating that module's own stated purpose ("no two drones lift from the same spot") and its BAY_SPACING_M = 35, chosen specifically to exceed the horizontal separation minimum so aircraft are deconflicted on the pad.

Fill: fan drones within a shared site along the pad's usable axis, respecting BAY_SPACING_M and the site's declared footprint.

LB-3Weather closure is index-keyed and fleet-globalenhancement

computeBayStatuses looks up bay-${i} while the weather engine generates exactly bay-0…bay-7 from a single boolean (gusts > 30 || ceiling < 200 || snow_ice). So a scenario with more than eight sites has the rest silently always-open, and every bay closes together. There is no notion of a sheltered pad — a parking structure, a rooftop with a windbreak, a vessel in the lee — versus an exposed one. That distinction is the entire tactical content of a weather bay decision.

Fill: exposure: 'sheltered' | 'semi' | 'exposed' on the site, with per-bay gust/ceiling thresholds. Key availability by siteId, not index.

LB-4Sites are keyed by droneId, so the "pool" is an illusionstructural

Both the catalog and the custom-mission compiler write launchSites[droneId] = …, and the planner seeds assignments with if (launchSites[id]) rec[id] = id. Launch sites are therefore effectively per-drone, not a shared pool — which makes capacityDrones nearly meaningless and makes the assignment dropdown misleading, since it offers UAV-05's site to UAV-02. The option labels render agency — kind and drop the site label and position entirely, so the operator chooses between "SFPD — police_rooftop" entries with no distance, no ETA, no capacity and no weather state.

Fill: give LaunchRecoverySite its own id, treat sites as a genuine shared pool, and render each option with distance, transit ETA, remaining slots, exposure and doctrine fit.

LB-5Mobile stations exist in the type system but cannot movethe ask

LaunchRecoverySiteKind already includes mobile_command, field_icp and vessel — three genuinely mobile platforms — yet scenario.launchSites is static configuration with no reposition path anywhere in the codebase. A mobile command post that cannot be moved is a rooftop with worse optics.

Fill: §D.3 — an override layer, map-drag repositioning, relocation time, and full evidence logging.

LB-6No recovery-side symmetry and no round-trip feasibilityenhancement

The planner never surfaces or validates recoverySites, even though buildSafeDroneRoutes silently appends an "RTB Safe Recovery" waypoint to a base the operator never saw. Nothing checks that the assigned bay is within round-trip range given platform endurance, reserve and weather drain — the exact arithmetic rechargeStations.ts already implements. A drone can be assigned a bay it cannot fly the mission from and return.

Fill: reachability becomes a hard gate in the doctrine scorer, and the planner shows the launch → mission → recovery triangle with margin.

D.3 Mobile stations the operator can reposition

Type additions

interface LaunchRecoverySite {
  id: string                       // NEW — sites become a real pool, not droneId-keyed  (LB-4)
  kind: LaunchRecoverySiteKind
  // …existing fields…
  exposure?: 'sheltered' | 'semi' | 'exposed'   // NEW — per-bay weather gating          (LB-3)
  mobile?: boolean                 // NEW — derived default: kind ∈ {mobile_command, field_icp, vessel}
  repositionRadiusM?: number       // NEW — how far it may credibly relocate
  repositionTimeSec?: number       // NEW — how long the move takes (vehicle transit + set-up)
  padFootprintM?: number           // NEW — usable pad width, feeds the intra-site fan    (LB-2)
}

State — an override layer, never a scenario mutation

droneStore.siteOverrides : Record<siteId, LatLng>
resolveLaunchSite(scenario, siteId, overrides) → LaunchRecoverySite

Why an override layer instead of editing the scenario

ScenarioConfig must stay immutable: it is the deterministic input that a seed reproduces, it is what the classroom broadcasts to every student byte-identically, and it is what replay and the after-action package reference. Repositioning writes to a separate override map, and one resolver replaces every direct scenario.launchSites[...] read — there are nine such sites in the codebase today.

Interaction — reuse the two editing gestures that already exist

  • Desktop: drag the site marker, using the same draggable-marker pattern the route editor already implements.
  • Mobile: tap-to-place, using the existing touch-edit map-click handler.
  • Constraints, live: clamped to repositionRadiusM, rejected inside an active geofence, and shown with a live delta — "−1.4 km to sector · +9% reserve margin · 4 drones re-planned."
  • Fixed sites are not draggable. A police rooftop does not move; the affordance must be absent, not merely disabled, or the model teaches the wrong thing.

Mid-mission relocation is an operation, not a teleport

Before launch, repositioning is free. After launch it takes time and has consequences, which is precisely what makes it a tactical decision:

  1. The site enters a relocating state for repositionTimeSec and accepts no recoveries during transit.
  2. Drones currently RTB'ing to that site must be re-vectored — Track A handles this natively as a "base moved" trigger, and the advisor proposes the re-plan the instant the move commits.
  3. A drone below reserve with no reachable alternative blocks the move, with the reason surfaced. You cannot strand an aircraft by driving away from it.
  4. A launch_site_repositioned event enters the chain of custody with from, to, affected drones and the reserve-margin delta.

GTests & enforcement

Two mechanical guarantees, pulling in opposite directions

Parity — what must be in both shells. parityMatrix.spec.ts walks both shells' actual import graphs and asserts every capability is reachable from each. Adding handleFleetRetask and handleRepositionSite to ACTION_HANDLERS means the build fails if either lands in only one shell. MissionScorecard is deliberately not a parity row — it is classroom-only, and adding it would assert exactly the leak §1.3 forbids.

Isolation — what must be in neither shell. Largely solved already: classroomBundleGuard.spec.ts proves WebSocket confinement and the flag-gated dynamic import at the source level, and scripts/assert-bundle-isolation.mjs proves it on the built artifacts in CI. The two assertions below close the single remaining hole described in §1.3 — a statically imported, non-networking classroom module, which is exactly what track C introduces.

it('keeps classroom vocabulary out of shared types', () => {

IRisks — what will actually bite

  1. The attribution trap silently corrupts everything downstream. If instructor commands are not distinguished from student commands, the rubric measures the wrong person and every debrief built on it is wrong. Fix it in the same commit as the command channel, never after.
  2. The sim's own guards run first. An advisor — or an instructor command — that ignores reserve, geofence and weather thresholds produces plans the simulation overrules a tick later. Indistinguishable from a bug at a glance.
  3. Lexicographic, not weighted. The moment life safety becomes a weighted term the rubric is gameable and the product's ethical claim evaporates. Keep it a cap.
  4. Undo must ship with the button. An instant fleet apply at 20× speed with no undo is a bad afternoon for a student who mis-taps.
  5. Visible takeover is not optional in a classroom product. Silent screen control is a trust and policy failure regardless of how well it is encrypted.
  6. Repositioning must never mutate ScenarioConfig. That object is the deterministic input a seed reproduces and the bytes the classroom broadcasts. Overrides only.
  7. Fixed sites must not appear draggable. A police rooftop does not move; an affordance that is present but disabled teaches the wrong mental model.
  8. Never fork decision logic into a shell. Sim, store and hook own everything; shells render. The parity matrix is what catches a violation.
  9. Classroom leakage arrives quietly, through shared surfaces. Not through an obvious import — through a types/index.ts union member, an after-action export field, or a "harmless" parity row. Each looks like a one-line convenience. Together they put the teaching edition inside the two operator editions. The isolation guard in §G is what makes the invariant survive contact with future work.
  10. Do not let the fidelity principle be over-applied. "Uniform simulation across builds" is about physics and determinism. It is not a licence to ship a grading rubric to an operator who has no instructor — an inversion this plan made in an earlier draft and has since corrected.
